# start_workspace — reference detail

Factored out of `start_workspace.md`. The body skill is authoritative for the
flow; this file holds the verbatim impact-report and registry formats.

## Linked-mode API-change impact

When a "Library PR Created" comment exists on the issue, fetch the upstream
library PR body and parse its `## API Changes` summary plus the `<details>`
block (Removed / Added / Renamed / Changed Signature / Changed Behaviour /
Migration). For each removed/renamed/changed symbol, grep the workspace scripts:

```bash
grep -rn "<old_symbol>" \
  autofit_workspace/scripts/ autogalaxy_workspace/scripts/ \
  autolens_workspace/scripts/ autolens_workspace_test/scripts/
```

Present an impact report listing the upstream PR, the API-change summary, the
affected scripts grouped by workspace, and the migration (before/after) lifted
from the upstream PR. If grep finds **no** affected scripts, tell the user this
may be the `/ship_library` option-(iii) path — offer to run `/smoke_test` and
merge the library PR instead of doing workspace edits.

## Attaching workspace repos to an existing worktree (linked mode)

The task already has a worktree root from `/start_library`, with workspace repos
present as symlinks. Replace each affected workspace symlink with a real worktree
on the same branch:

```bash
source admin_jammy/software/worktree.sh
worktree_add_repo <task-name> autofit_workspace      # one call per affected workspace repo
```

`worktree_add_repo` is idempotent (no-ops if already a real worktree); the
feature branch is created from `origin/main` on first attach. All edits/commits/
runs happen inside `$WT_ROOT/<workspace_repo>`.

## Creating a worktree for a standalone workspace task

```bash
source admin_jammy/software/worktree.sh
worktree_create <task-name> <workspace_repo1> [workspace_repo2 ...]
```

Same semantics as the library variant (real worktrees on `feature/<task-name>`,
symlinks for everything else, `activate.sh` written).

## pending-release label guard

Before `/ship_workspace` runs, assert the canonical `pending-release` label so
the later `gh pr create --label pending-release` cannot silently fail:

```bash
bash admin_jammy/software/ensure_workspace_labels.sh
```

It is idempotent (no-op when nothing drifted) and bootstraps the label where
missing. If it exits non-zero, stop and surface the failure.

## active.md registration

Linked mode — keep the `worktree:` from `/start_library`, extend `repos:` with
the workspace repos, set `status: workspace-dev`, add `library-pr:`:

```markdown
## <task-name>
- issue: <issue-url>
- session: claude --resume <session-id>
- status: workspace-dev
- worktree: ~/Code/PyAutoLabs-wt/<task-name>
- library-pr: <PR URL>
- repos:
  - PyAutoFit: feature/<task-name>
  - autofit_workspace: feature/<task-name>
```

Standalone mode — same shape without `library-pr:` and with only workspace repos.
Push with `prompt_sync_push "prompt: register <task-name> workspace repos in active.md"`.

## "Ready to develop" summary

Show: issue (+ upstream PR if linked), worktree root, the activation reminder,
the workspace repos attached (edit these, not the main checkout), the scripts to
update/create (with before/after for migrations), and the rules — only edit
`scripts/`, never `notebooks/` directly (regenerated by `/ship_workspace`). End
with "When done, run `/ship_workspace` to validate and create PRs."
